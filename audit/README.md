# CA Biositing Database Audit System

This directory contains a standardized, reproducible audit system for the CA
Biositing project. It is designed to provide a holistic overview of the
database's data quality, coverage, and integrity across AIM1 (analytical) and
AIM2 (processing) domains.

## Directory Structure

- `main.py`: The Python orchestrator that runs the audit using
  `ca_biositing.datamodels` utilities.
- `sql/`: Granular SQL modules for each record type, including lineage and
  quality checks.
- `AGENTS.md`: Specific instructions for AI agents to interpret the audit
  results.
- `REPORT_PROMPT.md`: A template and prompt for generating a comprehensive audit
  report.
- `output/`: (Gitignored) Contains the generated audit artifacts.
  - `audit_data.json`: Structured data for programmatic use.
  - `audit_summary.md`: Compiled raw output for LLM ingestion.

## Usage

### Local Environment

Run the audit against your local Docker-hosted database:

```bash
pixi run audit
```

_Note: This command automatically sets `POSTGRES_HOST=localhost` to connect to
your local services._

### Staging Environment

Run the audit against the Staging database (requires the Cloud SQL Auth Proxy
tunnel to be active):

```bash
pixi run audit-staging
```

## Adding New Audits

To add a new audit check:

1. Create a new `.sql` file in the `sql/` directory.
2. The script will automatically pick it up and include it in the next run.
3. If specific interpretation logic is needed, update `AGENTS.md` and
   `REPORT_PROMPT.md`.

## Materialized View Alignment

The `data_portal` schema contains 9 materialized views optimized for the
frontend. These views must align with the [MVP Specification](20260505_mv_spec).

### 1. Identify Mismatches

Run the standard audit to check if counts or basic structures look incorrect:

```bash
pixi run audit
```

Compare the output against the `20260505_mv_spec` requirements.

### 2. Update SQLAlchemy Definitions

Modify the view definitions in
`src/ca_biositing/datamodels/ca_biositing/datamodels/data_portal_views/`. These
files define the business logic using SQLAlchemy `select()` expressions.

### 3. Compile Migration

Because materialized views are managed manually (not autogenerated by Alembic),
use the compilation script to generate a raw SQL migration:

```bash
pixi run python scripts/compile_mv_fixes.py
```

This script:

- Compiles SQLAlchemy expressions into PostgreSQL-compatible SQL.
- Identifies required indexes (including `UNIQUE` indexes for concurrent
  refresh).
- Generates `alembic/versions/0100_mv_view_fixes.py` with `DROP/CREATE` logic.

### 4. Apply and Refresh

Apply the migration and refresh the data:

```bash
# Apply schema changes (may need to downgrade first if iteration is needed)
POSTGRES_HOST=localhost pixi run migrate

# Re-populate the views
POSTGRES_HOST=localhost pixi run refresh-views
```

### 5. Verification

Verify the views are correctly populated and indexed:

```bash
POSTGRES_HOST=localhost pixi run access-db -c "\d+ data_portal.mv_biomass_search"
```
